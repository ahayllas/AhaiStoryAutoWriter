# Execution rules
- "segment_objective" must concretely execute the target segment blueprint, not replace it.
- "required_beats" must be defined for this segment according to the blueprint. Each beat should be specific, concrete, and directly support the segment_objective
- "narrative_role" should align with the blueprint's segment_role.
- "scene_moves" should be in causal order.
- "must_include" should reflect assigned mandatory events and key blueprint constraints.
- "avoid" must include repetition and premature-resolution risks relevant to this segment.
- "ending_state_goal" should align with the blueprint's expected_end_state unless adaptation is necessary.
- If "must_include" contains a mandatory event, "state_update_expectations" must include the resulting factual change or consequence after that event happens.

# Adaptation rules
- You may adapt details based on current story state and current section progress.
- But do not invent a new structural purpose for this segment.
- If a blueprint-assigned event appears already completed, convert this segment into consequence, fallout, or escalation rather than repeating the event.
- If a previously assigned mandatory event was missed and must be recovered now, recover it in a way that still preserves distinct segment function.

# Anti-failure rules
- Do not produce vague beats like "tension rises".
- Do not propose a segment that only repeats existing information.
- Do not ignore the target segment blueprint.
- Do not add major new subplots in a closing segment unless necessary for payoff.
- Do not resolve the entire story in a non-final section.
- Do not leave the segment without a meaningful state change.
